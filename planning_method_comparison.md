# 赛车线路规划器对比分析：ECOS vs. Acados (SQP)

本报告深入研究了 `local_racing_line_planner` 的实现，并将其与之前在 `traj_planner.cpp` 中使用的基于 ECOS 的方法进行了对比。重点分析了 GGV 约束的实现方式以及求解速度差异的原因。

---

## 1. 对比概览表

| 对比维度                 | ECOS 方法 (`traj_planner.cpp`)  | Acados 方法 (`local_racing_line_planner.py`) |
|:-------------------- |:----------------------------- |:------------------------------------------ |
| **GGV 建模方式**         | **分段线性近似 (Piecewise Linear)** | **全非线性约束 (Nonlinear)**                     |
| **约束平滑度**            | 会在“转角”处产生不连续的梯度               | C1 连续的平滑边界                                 |
| **求解器算法**            | 通用内点法 (SOCP)                  | 专门优化的序列二次规划 (SQP)                          |
| **热启动 (Warm-start)** | 有限/基本没有                       | 高效重用上一帧的状态和乘子                              |
| **计算复杂度**            | $O(N^3)$ (通用求解器)              | **$O(N)$** (利用 Riccati 递归的块结构)             |
| **路径与速度耦合**          | 解耦进行 (先路径后速度)                 | **全耦合优化** (同时求解 $s, V, n, \chi$ 等)         |
| **工程实现**             | 手动构建 CCS 稀疏矩阵 (容易出错)          | 通过 CasADi 自动生成高度优化的 C 代码                   |

---

## 2. GGV 非线性约束的实现

核心区别在于车辆物理极限（G-G-V 图）的数学表达方式。

### ECOS 方法 (分段线性近似)

在 `traj_planner.cpp` 中，GGV 区域被简化为一组 $(v^2, a_t)$ 空间下的**线性不等式**。

```cpp
// 摘自 traj_planner.cpp: VelOptGlobal
G(LPnum * i + 2, i * nv + 0) = -Cw2_1; // v^2 的系数
G(LPnum * i + 2, i * nv + 1) = -1;      // at 的系数
h(LPnum * i + 2) = Cw0_1;              // 常数
// 结果等效于: -Cw2 * v^2 - at <= Cw0  =>  at >= -Cw0 - Cw2 * v^2
```

* **逻辑**：通过几条直线（超平面）手动“框出”GGV 的包络。
* **局限性**：这是一种**保守的近似**。为了保证安全，往往牺牲了一部分车辆性能。此外，线性分段的转角会导致控制量出现跳变。

### Acados 方法 (直接非线性约束)

在 `point_mass_model.py` 中，GGV 被视为真正的**非线性边界**。

```python
# 摘自 point_mass_model.py: diamond 模式
model.con_h_expr = ca.vertcat(
    ax_max - ax_tilde,
    ay_max - ca.fabs(ay_tilde),
    ca.fabs(ax_min) * ca.power(
        (1.0 - ca.power(ca.fabs(ay_tilde) / ay_max, gg_exponent)),
        1.0 / gg_exponent
    ) - ca.fabs(ax_tilde)
)
```

* **逻辑**：使用幂函数耦合公式（$|a_x/a_{max}|^P + |a_y/a_{max}|^P \le 1$）或直接使用 CasADi 的极坐标插值（Polar Interpolation）。
* **优势**：在全速度范围内保持**平滑且精确**的包络。求解器可以在不留保守余量的情况下，直接把车辆推向物理极限。

## 3. 为何 Acados 求解速度极快且效果更好？

之前的 ECOS 方法效果不佳，主要是受限于其求解器架构和问题构建方式。

### A. 专门的 SQP 算法与热启动

* **ECOS (SOCP)**：通用求解器，将每次求解视为独立任务，无法有效利用赛车运动的“连续性”。
* **Acados (SQP)**：利用上一帧的解作为**初始猜测 (Warm-start)**。由于赛车轨迹变化平滑，SQP 通常只需几次（甚至 1 次，即 RTI 模式）迭代即可收敛。

### B. 结构化求解 (Riccati 递归)

* **ECOS**：你手动构建了巨大的 CCS 稀疏矩阵。在这种方式下，求解器丢失了轨迹优化问题的“时间序列”结构（Markovian property）。
* **Acados**：使用 **HPIPM** 求解器，其内部使用了 **Riccati 递归**。对于轨迹长为 $N$ 的问题，其计算量与 $N$ 成正比（$O(N)$），而通用稀疏求解器的复杂度通常更高。

### C. 路径与速度的全耦合

* **旧方法**：通常先优化路径（PathOptLocal）再优化速度（VelOptGlobal）。这种**解耦**会导致路径规划器在挑选线路时，对后续速度规划的物理限制缺乏感知，容易产生无效解。
* **新方法**：Acados 同时求解所有状态。求解器知道：改变横向偏移（$n$）会改变曲率，进而直接触发非线性 GGV 约束对速度（$V$）的限制。

## 4. Acados 求解方法全方位技术细节

本节将从数学和工程角度，详细说明 `local_racing_line_planner` 中 Acados 求解器的具体配置。

### A. 目标函数 (Objective Function)

求解器的核心目标是 **时间最优 (Time Optimality)**。由于问题是在 $s$ 域（空间域）建模的，时间 $T$ 的积分被转化为：
$$\min \int_0^S \frac{1}{\dot{s}} ds$$
在代码 (`point_mass_model.py`) 中，代价函数定义为：

```python
model.cost_expr_ext_cost = 1.0 / s_dot + weight_jx * (jx / s_dot)**2 + weight_jy * (jy / s_dot)**2
```

* **项 1 ($1/\dot{s}$)**：直接最小化通过一段距离所需的时间。
* **项 2 & 3 (Jerk 惩罚)**：对纵向加加速度 ($j_x$) 和横向加加速度 ($j_y$) 施加惩罚，以保证轨迹的平滑性，防止控制器指令剧烈跳变。

### B. 状态变量与动力学约束 (Dynamics)

系统包含 7 个状态变量：$x = [s, V, n, \chi, a_x, a_y, t]^T$。
动力学模型采用了 **Frenet 坐标系下的点质量模型**，其微分方程组（对 $s$ 求导）考虑了赛道的曲率 $\Omega_z$。

* **空间域演化**：所有导数都乘以了 $1/\dot{s}$，将时间域转化为空间域。
* **二阶耦合**：加速度 $a_x, a_y$ 作为状态量，其变化受控制量 Jerk 驱动，这种三阶建模方式保证了曲率和速度曲线的连续性。

### C. 约束条件 (Constraints)

Acados 高效处理了三类关键约束：

1. **初始状态约束 (Initial Bounds)**：强制 $x_0$ 等于赛车当前传感器的真实状态。
2. **路径走廊约束 (Path Constraints)**：
   * $n_{min} \le n \le n_{max}$：保证赛车在赛道边界（或由 Carver 定义的可行域）内。
   * 采用了 **软约束 (Slacks)** 技术，允许在极少数情况下轻微违越边界以保证求解不崩溃。
3. **非线性 GGV 约束 (GGV-Diagram)**：
   * 在 `diamond` 模式下，使用耦合公式：$\left(\frac{|a_x|}{a_{x,max}}\right)^P + \left(\frac{|a_y|}{a_{y,max}}\right)^P \le 1$。
   * 这是求解器性能压榨最充分的地方，直接利用了 CasADi 生成的 Jacobian 和 Hessian 信息进行精确求解。

### D. 离散化方法 (Discretization)

Acados 采用了 **多重射击法 (Multiple Shooting)**：

* **积分器**：代码中配置为 `ERK` (Explicit Runge-Kutta 4th order)。4 阶龙格库塔法在保证精度的同时，提供了极高的计算效率。
* **网格划分**：通常将预测时域（例如 300-500 米）划分为 $N=150$ 个阶段。这种离散化将无限维的连续最优控制问题转化为有限维的非线性规划问题 (NLP)。

### E. 求解算法 (Solution Strategy)

这是 Acados 速度远超 ECOS 的技术护城河：

1. **SQP (序列二次规划)**：将非线性问题在当前迭代点线性化，转化为一个二次规划 (QP) 问题。对于赛车运动，上一帧的解为当前帧提供了极佳的 **热启动 (Warm-start)** 点，因此通常 1-2 次迭代即可收敛。
2. **HPIPM 求解器**：底层的 QP 求解器是专门为最优控制设计的 HPIPM。它利用了 **Riccati 递归** 算法，将求解复杂度从通用求解器的 $O(N^3)$ 降低到了 $O(N)$。
3. **C 代码生成**：Acados 通过 CasADi 将复杂的模型方程提前编译为高度优化的 C 代码，避免了 Python 运行时的开销。

### F. 规划时域与终端约束 (Horizon & Terminal State)

这是一个非常关键的问题：Acados 既然是“局部”规划，它是如何确定优化多远的距离，以及终点在哪里的？

1. **规划时域 (Planning Horizon)**：
   
   * 在 Acados 中，这是通过 `solver_options.tf` 参数确定的。
   * **关键点**：由于我们的模型是以 $s$（里程）为自变量的，这里的 `tf` 代表的是**总里程**（例如 300 米或 500 米），而不是总时间。
   * 通过将总里程 `tf` 除以步数 `N` (例如 150 步)，我们就得到了空间离散化的步长 $\Delta s = 2 \sim 3.3$ 米。这保证了无论车速快慢，规划器始终能覆盖前方固定距离的赛道信息。

2. **终端状态约束 (Terminal Constraints)**：
   
   * 在当前的实现中，终端状态 $x_N$ 是 **自由 (Free)** 的，即没有设置硬性的终端约束（如必须回到赛道中心或达到特定速度）。
   * **为什么不需要？** 
     * **收缩时域性质 (Receding Horizon)**：我们采用的是 MPC（模型预测控制）思想，每一帧只执行规划结果的第一步，然后下一帧立即重规划。终端状态的微小偏差对当前的动作影响极小。
     * **代价函数驱动**：由于代价函数中包含 $1/\dot{s}$（时间最优），求解器自然会尝试在给定的里程内，以最快的速度、最合理的姿态到达终点。
     * **路径走廊约束**：虽然没有终端状态点约束，但整个预测区间（包括最后一个节点）都受到赛道边界约束的限制，保证了终点不会越界。

### G. GGV 数据的读取与处理 (GGV Data Handling)

Acados 能够如此高效地处理非线性 GGV 约束，离不开一套专门的 GGV 数据管理机制。

1. **数据格式**：
   GGV 数据并不是一个单一的文件，而是一个包含多个 `.npy`（NumPy）文件的文件夹。主要包含：
   
   * `v_list.npy`：速度断点（Breaks）。
   * `g_list.npy`：法向力/载荷因子断点（考虑下压力）。
   * `rho.npy`：极坐标下的最大加速度半径（用于 Polar 模式）。
   * `ax_max.npy`, `ax_min.npy`, `ay_max.npy`, `gg_exponent.npy`：用于 Diamond 模式的四个关键参数矩阵。

2. **读取与预处理 (GGManager.py)**：
   系统通过 `GGManager` 类统一加载这些数据，并在加载时进行了以下处理：
   
   * **边界扩展**：自动补齐 $V=0$ 和 $g=0$ 的数据点，确保规划器在极低速或极端载荷下也有定义。
   * **安全余量 (Margin)**：通过 `gg_margin` 参数（例如 0.05），将原始数据整体收缩 5%，为实际控制留出容错空间。

3. **CasADi 符号化插值**：
   这是最独特的地方。系统并没有在每步循环中手动查表，而是利用 **CasADi 的 `interpolant` 函数** 将离散的 NumPy 矩阵转化为了**连续可微的数学函数**。
   
   * **自动求导**：由于使用了符号化插值，Acados 在求解过程中可以自动获取加速度极限对速度 $V$ 的 Jacobian（雅可比矩阵），从而在寻找最优速度时知道“如果我快 1m/s，我的侧向抓地力会损失多少”。
   * **高效集成**：这些插值函数直接编译进了 Acados 的 OCP 模型中，执行速度极快。

### H. 在线自适应性 (Online Adaptability)

这是一个非常深刻的问题：既然 GGV 是提前“编译”进代码的，那么如果比赛中途下雨（摩擦系数改变），该表是否还能修改？

1. **静态编译的局限性**：
   
   * 由于 `interpolant` 生成的函数已经“硬编码”进了 CasADi 的符号计算图中，并在 Acados 初始化时生成了 C 代码，你**无法**在求解的瞬间动态修改 `.npy` 里的原始数据点。
   * 如果你想彻底更换一张表，目前只能通过销毁并重新初始化 `AcadosOcpSolver` 来实现，但这在 100Hz 的实时控制逻辑中开销过大。

2. **主流的自适应方法：摩擦系数缩放**：
   
   * 虽然不能改表，但通常的做法是在模型中引入一个 **参数 (Parameter)**（例如 $\mu$），并让 $a_{x,max} = \mu \cdot f_{lookup}(V, g)$。
   * Acados 支持 **Runtime Parameters**。这意味着你可以在每一帧调用 `solver.set(i, "p", mu)` 来改变当前的抓地力水平。
   * 在当前的实现中，系统主要通过 `gg_margin` 在初始化时进行缩放。如果需要实时的路面自适应，可以通过在 `point_mass_model.py` 中增加一个 `mu` 参数来完美解决。

---

**对比总结**：ECOS 版通过手动构建矩阵，理论上每帧都可以传一张新表，但在大型非线性问题下效率极低；Acados 版选择了“结构固定、参数可调”的方案，通过预编译非线性结构保证了极限性能，而将自适应性交给了参数缩放。
